Getting started

wardn runs entirely in your environment — no data leaves your network. Every integration is read-only by default, scoped to exactly what you grant, and you're in full control of what the agent can touch. Up in under ten minutes.

Prerequisites

You need the following on the host machine:

  • Docker 24+ — wardn ships as a set of containers. Verify with docker --version.
  • Docker Compose v2 — bundled with Docker Desktop; on Linux install the docker-compose-plugin package. Verify with docker compose version.
  • 8 GB RAM minimum available for the container set. 16 GB recommended for production use with multiple integrations.
  • An LLM API key — Anthropic Claude is recommended. Any OpenAI-compatible endpoint is supported, including locally-hosted models.
wardn needs no public IP and no inbound ports. All traffic is outbound — from your network to your tools.

Installation

1

Obtain container registry credentials

wardn images are hosted in a private container registry. Contact sales to receive your registry username and token, then authenticate Docker:

docker login ghcr.io -u YOUR_USERNAME --password-stdin
# Paste your registry token and press Enter
Registry credentials are required. docker compose up will fail to pull images without them. Reach out to sales@wardn.run if you have not yet received your token.
2

Clone the repository

Clone to the machine running wardn.

git clone https://github.com/wardn-run/wardn.git
cd wardn
3

Create your environment file

Copy the example and open it in your editor. At minimum set POSTGRES_PASSWORD, SECRET_KEY, and at least one LLM API key (e.g. ANTHROPIC_API_KEY).

cp .env.example .env
# Edit .env — set POSTGRES_PASSWORD, SECRET_KEY, and ANTHROPIC_API_KEY at minimum
4

Start wardn

Pulls images on first run — give it a minute.

make up
5

Verify the stack is healthy

All containers should be healthy within 30 seconds.

docker compose ps
# All services should show: running (healthy)

curl http://localhost:8000/health
# {"status":"ok"}
An LLM API key is required. wardn will start without one but the AI assistant will be disabled. Set at least one provider key (e.g. ANTHROPIC_API_KEY) before using any skill or the chat interface.
Deploying wardn on Kubernetes is coming soon. A Helm chart and production-grade K8s manifests are in active development. If you want early access or have an enterprise deployment requirement, contact sales.

First run

1

Open the console

Navigate to http://localhost:3000 in your browser.

2

Create a workspace

Click New workspace and pick a template — incidents, kubernetes, aws-infra, or start blank. Templates pre-load context and skills for that domain.

3

Connect an integration

Go to Settings → Integrations and add credentials for at least one tool. Datadog, Kubernetes, or Slack are the most useful starting points. All credentials are encrypted at rest.

4

Run your first skill

Type /morning in the workspace chat. wardn queries all connected integrations in parallel and returns a full ops briefing.

Dashboard

Your home screen. One view of everything that matters right now.

What changed

The top strip surfaces auto-detected anomalies ranked by impact, classified into four types:

  • Incidents — active alerts or monitors in a firing state across Datadog or ArgoCD.
  • Regressions — performance or reliability metrics that have moved meaningfully outside their historical range.
  • Cost drift — AWS or Kubernetes spend tracking above or below its 7-day baseline by a configurable threshold (default 20%).
  • Opportunities — positive signals such as cost savings from right-sizing, or services performing above their peer group.

Each item links to the source data and has a one-click Investigate action that opens a /investigate session scoped to that anomaly.

Cost tracking

Month-to-date spend with a 30-day sparkline, aggregated across connected cost sources — AWS, Datadog cloud cost metrics, and Kubernetes namespace costs via OpenCost. Updates hourly, delta vs the previous 30 days.

Deployments

Recent ArgoCD syncs with status (Synced, OutOfSync, Degraded) and health. Click any deployment for full detail.

Open tickets

High-priority issues assigned to you from your connected ticketing tool. Supports Linear and others. Hidden when no ticketing integration is configured.

Skills

Skills are the things you run in chat. Type a slash command — or just describe what you want — and wardn queries your integrations in parallel, runs the analysis, and returns structured output. Comes with built-ins for common ops workflows. Build your own or import from your team.

You don't need the slash command. "what happened overnight?" or "why is the payments service slow?" — wardn picks the right skill.

/morning

/morning /briefing Morning Briefing

Your shift start briefing. Hits every connected integration in parallel and returns a full overnight summary — ready to read or post straight to Slack.

Queries
Datadog monitors AWS Cost Explorer Kubernetes events Linear open tickets ArgoCD sync status
Output

HTML widget with overnight service health, cost deltas vs 7-day average, open alerts, assigned Linear tickets, and recommended actions ordered by urgency.

/cost

/cost Cost Analysis

Pulls spend from AWS, Datadog, and OpenCost, then cross-correlates cost spikes with deployments and scaling events. Tells you what changed and when.

Queries
AWS Cost Explorer Datadog cloud cost OpenCost via Grafana Kubernetes scaling events
Output

Spend breakdown by service, week-over-week trend chart, anomaly highlights with timestamp of inflection, and a root cause hypothesis linking cost changes to infrastructure events.

/handover

/handover Incident Handover

Reads your Slack incident channels and correlates with Datadog, Kubernetes, ArgoCD, and Linear. Comes out the other end as a structured handover doc, ready to post.

Queries
Slack incident channels Datadog incidents Kubernetes events & logs ArgoCD rollout history Linear tickets
Output

Structured handover card (Summary, Timeline, Current State, Actions Taken, Outstanding Items, Next Steps). Auto-posted to configured Slack channel and saved as a Notion page.

Default lookback is 8 hours. Pass a duration or timestamp to override: /handover 12h or /handover 2026-01-15T06:00Z.

/investigate

/investigate Broad Investigation

Something's wrong but you don't know where. Queries everything for the last 6 hours and builds a change correlation timeline.

Queries
Datadog monitors & metrics Kubernetes events Linear open issues ArgoCD sync history Runbook search
Output

Cross-source report with a unified change correlation timeline, relevant runbook matches, and a prioritized list of hypotheses. Each hypothesis links to the supporting evidence.

/k8s

/k8s /pod [name] Kubernetes Incident

Systematic K8s triage. /k8s sweeps the namespace; /pod [name] targets a specific pod. Works through status, events, logs, node pressure, and recent deploys in order.

Kubernetes pod status & events Pod logs (last 200 lines) Node resource pressure Datadog alerts ArgoCD rollout history
Output

Root cause analysis with the most likely failure reason, remediation steps, and optional one-click actions to create a Notion runbook or a Linear ticket.

Full post-mortem saved to Notion with timeline, 5-whys chain, contributing factors, and action items. Linear tickets are created automatically for each action item.

Building your own skills

Every skill is a SKILL.md file — plain Markdown, no code. Describe the trigger, the integrations, and the output format. Drop it in the skills/ directory and it's live immediately.

# skills/my-service-health/SKILL.md ## Trigger `/my-health`, `/service-health`, or keywords: "service status", "is everything ok" ## Description Check the health of our payment and auth services. Queries Datadog for error rates and latency, checks Kubernetes pod health, and summarizes open PagerDuty alerts. ## Integrations - Datadog (metrics: service:payments, service:auth) - Kubernetes (namespace: production) - PagerDuty (service IDs in workspace agent.md) ## Output format HTML status card with: service name, p99 latency, error rate (1h), pod health, open alerts. Color-coded by severity. ## Mode flash # or: ptc (for deep research with code execution)

Manage skills from Settings → Skills in the console, or drop folders directly into skills/ on the filesystem.

Importing community skills

Skills are plain files — share via Git, copy between instances, or keep a team repo. To pull one from a URL:

curl -o skills/weekly-report/SKILL.md \
  https://raw.githubusercontent.com/your-org/wardn-skills/main/weekly-report/SKILL.md

Skill modes

Set mode in the SKILL.md file:

  • flash — conversational mode; fast responses without a code execution sandbox. Good for status checks and simple queries.
  • ptc — deep research mode; runs in a full Python sandbox with access to all MCP tools. Use for analysis, report generation, and multi-step investigations.

Workspaces

A workspace is a scoped context for the agent — its own agent.md memory, chat history, and a specific set of integrations you control. Each workspace can only see and act on what you've explicitly given it access to. Nothing bleeds across workspace boundaries.

What workspaces store

The agent writes to agent.md during every session, building up:

  • Service and infrastructure context — what services this workspace covers, their normal operating ranges, and known quirks.
  • Historical incidents — what happened, when, and how it was resolved.
  • Performance baselines — the "normal" latency, error rate, or cost for the services in scope.
  • Runbook links from Notion or Confluence.
  • Key contacts — who owns which service.

The more you use a workspace, the sharper the agent gets — it learns what "normal" looks like for your environment.

Templates

Templates pre-load context and skills for a given domain. Built-in options:

incidents

Active and recent incident tracking. Integrates Slack, Datadog, Linear, and Notion. Best for /handover and /rca.

aws-infra

AWS cost and infrastructure monitoring. Pre-loaded with Cost Explorer queries and the cost spike automation.

kubernetes

Kubernetes cluster operations. Configured for pod health sweeps and the /k8s skill.

blank

Start from scratch. Add your own context to agent.md and pick which skills and integrations this workspace uses.

Creating a custom template

Save any workspace as a template from the workspace menu — agent.md, skill bindings, and integration config are all preserved.

Thread management

Each workspace supports multiple named threads. They share agent.md but have separate conversation histories — useful for keeping an active incident separate from routine morning checks.

Sharing

Generate a read-only share link from the workspace menu to hand off a briefing or handover doc to stakeholders without a wardn account. Links expire in 7 days and can be revoked anytime.

Per-workspace integration config

Each workspace can override Slack channels, Linear team, or Notion database — so incidents posts to a different channel than aws-infra. Configure in Workspace settings → Integrations.

Automations

Automations run skills on a schedule or when a metric crosses a threshold — no manual trigger needed. Output posts to Slack, Notion, or wherever you configure.

Trigger types

Cron schedule

Standard cron syntax. Runs in the configured workspace's context with full access to its integrations.

# Run /morning in the incidents workspace every weekday at 07:00 UTC
schedule: "0 7 * * 1-5"
workspace: incidents
skill: morning

Metric threshold

wardn polls the integration every 5 minutes and fires the skill when the condition is met.

# Fire /cost when AWS daily spend is 25% above the 7-day average
type: metric_threshold
metric: aws.cost.daily_spend
condition: percent_change_above
threshold: 25
workspace: aws-infra
skill: cost

Built-in templates

Automation templates for common workflows — use as-is or as a starting point:

Daily Morning Briefing

cron

Runs /morning every weekday and posts the output to your ops Slack channel.

0 7 * * 1-5

Cost Spike Alert

threshold

Fires /cost when AWS spend exceeds 25% above the 7-day rolling average.

aws.cost > +25%

Weekly Report

cron

Generates a week-in-review document every Monday and creates a Notion page.

0 8 * * 1

Custom skill, any trigger

cron

Point any automation at a custom skill from your skills/ directory — built-in and custom skills are interchangeable.

any schedule or threshold

Creating automations

Create automations in the console or edit config.yaml directly. Any skill — built-in or custom — works as the target.

# config.yaml
automations:
  - name: "My custom health check"
    schedule: "*/15 * * * *"    # every 15 minutes
    workspace: "incidents"
    skill: "my-service-health"  # your custom skill slug
    mode: "flash"

Execution history

Every execution is logged with full output, duration, and any errors. View at Automations → History. Logs kept 90 days. Toggle automations on/off without deleting them.

Suggested automations: wardn watches your manual skill usage and suggests automating repeating patterns — if you run /handover every morning, it'll offer to schedule it.

AI providers

wardn works with any major AI provider. Ships with Anthropic Claude by default. Switch providers in .env or let users bring their own keys through the Settings UI.

Cloud API providers

Set the key in .env and restart. LLM_DEFAULT_MODEL and LLM_FLASH_MODEL control which model runs by default.

ProviderEnv variableNotes
Anthropic (Claude) ANTHROPIC_API_KEY Default provider. Recommended for best results. Get a key at console.anthropic.com.
OpenAI (GPT) OPENAI_API_KEY Supports GPT-5.4 and family. Get a key at platform.openai.com.
Google (Gemini) GEMINI_API_KEY Supports Gemini 3.x models. Get a key at aistudio.google.com.
Groq GROQ_API_KEY High-throughput inference. Useful for flash/subagent roles. console.groq.com.
DeepSeek DEEPSEEK_API_KEY DeepSeek V3.2 reasoning model. platform.deepseek.com.
OpenRouter OPENROUTER_API_KEY Single key for 200+ models from multiple providers. Useful for testing or cost optimisation. openrouter.ai.
Cerebras CEREBRAS_API_KEY Ultra-fast inference (~3000 tps) for latency-sensitive subagent workloads. cloud.cerebras.ai.

Update LLM_DEFAULT_MODEL and LLM_FLASH_MODEL, then restart:

# Example: switch to OpenAI
OPENAI_API_KEY=sk-...
LLM_DEFAULT_MODEL=gpt-5.4-mini

OAuth providers (no API key required)

Connect in Settings → AI Provider → Connect account. No env vars needed — wardn stores a per-user OAuth token. Connected providers appear in the model picker automatically.

Cl

Claude (OAuth)

Connect your Claude.ai account directly. Uses the same Claude models as the API but billed through your existing Claude subscription rather than API credits.

Requires Claude Max or Team plan.
GP

ChatGPT Codex (OAuth)

Connect your ChatGPT account via the Codex API. Runs OpenAI models billed through your ChatGPT Plus or Pro subscription.

Requires ChatGPT Plus or Pro.

Local models (self-hosted)

Run models locally — on the same machine or anywhere on your network. No API keys. wardn reaches your host via host.docker.internal automatically.

Docker networking: make sure Ollama or LM Studio is bound to 0.0.0.0, not just 127.0.0.1 — otherwise wardn's containers can't reach it.
ProviderDefault portSetup
Ollama 11434 Install from ollama.com, pull a model (ollama pull llama3.3), then select Ollama in Settings → AI Provider. No env var needed.
LM Studio 1234 Install from lmstudio.ai, load a model, enable the local server, then select LM Studio in Settings → AI Provider.
vLLM 8000 Start vLLM with --host 0.0.0.0. wardn will connect at http://host.docker.internal:8000/v1. Select vLLM in Settings → AI Provider.

Per-user API keys (BYOK)

Users can enter their own provider key under Settings → AI Provider → Bring your own key. Encrypted at rest, used only for their requests — never overrides the system default. Useful when you want teams to own their own model costs.
